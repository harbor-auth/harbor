# Grant-ID foreign key on sessions (plan)

> **Dependency order:** depends on **`client-grant-persistence`** (the `grants`
> table must exist and be populated before sessions can reference it) and on
> **`session-ppid-seam`** (which sets `RefreshSession.GrantID` at consent time).
> This plan is the final lock-down of the revocation scoping model.

## Problem

`RefreshSession.GrantID` is always `""` today. The `TODO(grant-fk)` comment in
`internal/clients/sessions.go` (`buildCreateSessionParams`) documents the
runtime assertion that silently drops any non-empty `GrantID`:

```go
// TODO(grant-fk): when the sessions table gains a grant_id FK column,
// parse rs.GrantID here (it is always "" today — see RefreshSession.GrantID).
if rs.GrantID != "" {
    return db.CreateSessionParams{}, fmt.Errorf(
        "sessions: GrantID %q is not supported (no grant_id DB column yet)", rs.GrantID)
}
```

Without `grant_id` on the session row, `RevokeSessionsByUserClient` revokes
**all** sessions for a `(user_id, client_id)` pair when a theft signal fires
(§3.5). That is intentionally conservative — it over-revokes rather than
under-revokes — but it has two consequences:

1. **"Log out everywhere" is conflated with "revoke on theft".** Both paths hit
   the same revocation query today. When a user logs out of a single app, all
   sessions for that app are revoked (no per-grant scoping).
2. **Multi-grant scenarios (future §11.3) are impossible.** A user who has
   granted two different apps the same `(user_id, client_id)` scope can't
   independently revoke one.

The fix is to add `grant_id UUID NOT NULL REFERENCES grants(id)` to the
`sessions` table and thread `GrantID` through the entire write path.

## Proposed approach

### 1. DB migration

```sql
-- 0003_grant_id_fk.up.sql
ALTER TABLE sessions
  ADD COLUMN grant_id UUID NOT NULL REFERENCES grants(id);
CREATE INDEX sessions_grant_id_idx ON sessions (grant_id);
```

Down migration drops the column and index. The column is `NOT NULL`, so a
backfill or a two-step migration (add nullable → backfill → set NOT NULL) is
needed if sessions already exist in production; for the current dev-only state
a single-step migration is acceptable.

### 2. sqlc query update

Add `grant_id` to `db/queries/sessions.sql`:
- `CREATE SESSION` — include `grant_id` in the `INSERT`.
- `REVOKE SESSIONS BY GRANT` — new query: `DELETE FROM sessions WHERE grant_id = $1`
  for targeted grant revocation (logout from one app).
- `REVOKE SESSIONS BY USER CLIENT` (theft signal) — remains broad by design
  (revoke **all** sessions for the family), so no change needed.

Regenerate `internal/gen/db/` via `sqlc generate`.

### 3. `buildCreateSessionParams` unblock

Remove the `GrantID != ""` guard and the `TODO(grant-fk)` comment. Map
`rs.GrantID` to `pgtype.UUID` alongside the existing `UserID` mapping:

```go
var grantID pgtype.UUID
if err := grantID.Scan(rs.GrantID); err != nil {
    return db.CreateSessionParams{}, fmt.Errorf("sessions: parse grant ID %q: %w", rs.GrantID, err)
}
```

### 4. `oidc.SessionStore` interface extension

Add `RevokeSessionsByGrant(ctx context.Context, grantID string) error` alongside
the existing `RevokeSessionsByUserClient`. Both `InMemorySessionStore` and
`DBSessionStore` must implement it. The in-memory store filters by `grantID`;
the DB store issues the new sqlc query.

### 5. Wire into service.go

`Refresh()` and `signalRefreshReuse()` already have access to
`session.GrantID`. Thread it through `RotateSession` so the new session row
carries the correct FK.

### 6. `service.go` comment cleanup

Remove the `// always copies "" (no DB column yet); see TODO(grant-fk)` note
from the `GrantID` assignment in `Refresh()`.

## DESIGN alignment

Realizes the `sessions → grants` FK relationship implied by §10 (the data
model). Enables the §11.3 "revoke a connected app" user flow (which needs
`RevokeSessionsByGrant` to avoid revoking unrelated sessions). Tightens §3.5
theft-signal scoping so future multi-grant scenarios work correctly. Does
**not** change `DESIGN.md`.
